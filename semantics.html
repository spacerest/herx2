general notes

    semantics overview

  • semantics in html describes the value of content on a page, not taking style or appearance into consideration
  • accurate semantic code helps out search engines, screen readers, etc

    elements that don’t hold semantic value — for stylistic purposes only

  • < div > — block–level element
  • < span > — inline–level element

    block– versus inline–level elements versus both

  • block–level elements begin on a new line, stack on top of each other, occupy any available width, sometimes nest inside each other, can wrap inline&ndashlevel elements
  • inline–level elements don’t start on new lines — they line up after one another & maintain the width of their content, sometimes nest inside each other, cannot wrap block–level elements, usually contain a couple of words

    some choices

  • choose heading ranks based on relevancy to the content of the page — not for sizing purposes
  • name classes and ids based on content, not desired appearance (desires change!)
  • < strong > inline–level element gives importance to text, while < b > is used to stylistically offset text
  • < em > should both emphasize and italicize (a new voice), while < i > should just italicize (i.e. a latin word for cat is feline

    some semantic structure choices—block–level elements

  • < header >
  • < nav >
  • < section >
  • < aside >
  • < footer >
  • < article >

    some (more?) inline–level elements that give structure and style

  • < small >
  • < a > — with hmtl5, these can now wrap block–level elements despite being inline–
